# Idea Log - 2026-03-14

## Problem Statement
We need a structured collaboration tool that continuously captures assistant-generated ideas, summarizes the problem/solution daily, and keeps the human partner in the loop without waiting for input from them.

## Idea
- Build a lightweight Idea Pipeline helper that:
  - Automatically records each idea I generate, tagging it with the current time and context.
  - Includes a daily summary section that distills the problem, proposed solution, and UX/test considerations.
  - Lives inside `ideas-generated-ai` so you can approve before implementation.
  - Notes whether approval has been given, and once it is, prepares the scaffolding for a new idea directory with implementation notes, mockups, and tests.

## UX & Testing Focus

- Add a quick Vitest check verifying that the latest daily summary file exists and includes the required metadata (title, problem, status).
- Consider a simple CLI helper (`npm run idea-summary`) to regenerate the report and run the quick verification so we’re always thinking about the user experience.

## Additional Idea: Collaborative Problem Mapper

### Problem Statement
We need a way to make each idea less abstract by mapping how it relates to the broader collaboration context—who will consume it, what dependencies it has, and how it might evolve—without waiting for the other person to prompt inputs.

### Idea
- Add a lightweight “Problem Mapper” section to the daily log that:
  - Lists the stakeholders (assistant, human, other teams) the idea touches and the action each is expected to take.
  - Captures dependencies (APIs, UX decisions, CI checks) that would block progress if not addressed.
  - Suggests a small experimental step (mockup, quick prototype, or set of tests) we can run once the idea is approved.

### UX & Testing Focus
- Format the mapper as an accessible table or bullet list so the human can scan it quickly during their daily review.
- Include a test helper that verifies the mapper section has at least one stakeholder and one dependency listed before the idea is marked for implementation.
- Consider a short script to render the mapper as markdown + JSON so we can one-click paste into GitHub issues or planning tools.

### Next Steps
1. Capture this mapper alongside new ideas in the daily log so approvals can include decisions about responsibilities and blockers.
2. Once you give the green light, spin up a directory like `ideas-generated-ai/problem-mapper` with a README, mapper template, and tests that exercise the helper scripts.
3. Reflect on the experience to make the mapper more conversational and less paperwork-y—maybe add emoji-style status tags for each dependency.

## New Idea: Reflection Reminder

### Problem Statement
Even with daily idea logs, we sometimes skip reviewing what worked previously or checking if previous experiments still make sense; without a simple reminder, these lessons can slip through the cracks.

### Idea
- Append a brief “Reflection Reminder” section to today's log that asks:
  - What part of yesterday's idea worked well and deserves repeating?
  - What assumption from the last experiment should we re-evaluate before we build more?
  - Are there any new safety/ethics checks we should consider before implementing the next module?
- Record one sentence answer per prompt so the reminder stays lightweight.

### UX & Testing Focus
- Keep the prompts short and bolded for quick scanning, and show the timestamp for when the reminder is written.
- Create a tiny test that ensures the reflection section exists and includes all three prompts before the daily log is approved.
- Offer a helper script that can print yesterday's answers so we can remind ourselves during the next daily sync.

### Next Steps
1. Include the reflection answers in today's log and keep them updated whenever you review older ideas.
2. After approval, add a folder like `ideas-generated-ai/reflection-reminder` with templates and scripts that surface these prompts automatically.
3. Evaluate how the reminder affects downstream work—does it prevent us from repeating mistakes or missing dependencies?
